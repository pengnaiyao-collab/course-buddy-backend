# 常见错误及解决方案

## 1. 概述

本文档汇总 Course Buddy Backend 运维和开发过程中常见的错误及其解决方案，涵盖应用启动、数据库连接、JWT 认证、WebSocket、MinIO 文件上传、Redis 缓存和日志分析等方面。

---

## 2. 应用启动问题

### 2.1 应用无法启动（Exit Code 1）

**症状**：
```
docker logs course-buddy-app
# 或
java -jar app.jar
# 显示 APPLICATION FAILED TO START
```

**原因一：数据库连接失败**
```
Error creating bean with name 'dataSource':
Failed to obtain JDBC Connection: Communications link failure
```

**解决方案**：
```bash
# 1. 检查 MySQL 是否运行
docker compose ps mysql
# 或
systemctl status mysql

# 2. 检查数据库连接参数
docker exec course-buddy-app env | grep DB_

# 3. 手动测试数据库连接
docker exec course-buddy-mysql mysql \
    -u ${DB_USER} \
    -p${DB_PASSWORD} \
    -e "SELECT 1" ${DB_NAME}

# 4. 检查防火墙（Linux）
sudo ufw status
sudo iptables -L | grep 3306

# 5. 确认 docker-compose.yml 中 app 依赖 mysql 健康检查
# depends_on:
#   mysql:
#     condition: service_healthy
```

**原因二：Redis 连接失败**
```
Cannot connect to Redis; nested exception is io.lettuce.core.RedisConnectionException
```

**解决方案**：
```bash
# 1. 检查 Redis 是否运行
docker compose ps redis

# 2. 测试 Redis 连接
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} ping
# 预期返回：PONG

# 3. 检查 REDIS_HOST 环境变量
# Docker Compose 内部应使用服务名 "redis"，不是 localhost
echo "REDIS_HOST 应为 redis (非 localhost)"
```

**原因三：JWT_SECRET 未配置或长度不足**
```
IllegalArgumentException: JWT secret key must be at least 32 characters
```

**解决方案**：
```bash
# 检查 JWT_SECRET
docker exec course-buddy-app env | grep JWT_SECRET

# 生成新的 JWT_SECRET（64字符随机字符串）
openssl rand -base64 64
# 将生成结果填入 .env 文件
```

**原因四：端口被占用**
```
Web server failed to start. Port 8080 was already in use.
```

**解决方案**：
```bash
# 查找占用 8080 端口的进程
lsof -i :8080
# 或
ss -tlnp | grep 8080

# 杀掉占用进程（根据 PID 操作）
kill -9 <PID>

# 或修改应用端口
# 环境变量：SERVER_PORT=8081
# application.yml: server.port: ${SERVER_PORT:8080}
```

### 2.2 Flyway 迁移失败

**症状**：
```
FlywayException: Validate failed:
Migration checksum mismatch for migration version 5
-> Applied to database : 1234567890
-> Resolved locally    : 9876543210
```

**原因**：已执行的迁移脚本文件内容被修改  
**解决方案**：
```bash
# 方案1（推荐）：恢复文件到原始内容
git show HEAD~1:src/main/resources/db/migration/V5__xxx.sql > \
    src/main/resources/db/migration/V5__xxx.sql

# 方案2（开发环境）：修复 checksum
mvn flyway:repair

# 方案3（开发环境）：清空重建
mvn flyway:clean flyway:migrate
# ⚠️ 警告：将删除所有数据，仅用于开发环境！
```

**症状二**：
```
Found more than one migration with version 10
```

**解决方案**：
```bash
# 检查是否有重复版本号的迁移文件
ls src/main/resources/db/migration/ | grep "^V10"
# 重命名其中一个文件
mv src/main/resources/db/migration/V10__xxx.sql \
   src/main/resources/db/migration/V10.1__xxx.sql
```

### 2.3 MinIO Bucket 初始化失败

**症状**：
```
ErrorResponseException: The specified bucket does not exist
```

**解决方案**：
```bash
# 1. 检查 MinIO 是否运行
curl http://localhost:9000/minio/health/live

# 2. 手动创建 Bucket
docker exec course-buddy-minio \
    mc alias set local http://localhost:9000 ${MINIO_ACCESS_KEY} ${MINIO_SECRET_KEY}
docker exec course-buddy-minio \
    mc mb local/course-buddy

# 3. 检查 MINIO_ENDPOINT 格式（必须含 http:// 前缀）
echo $MINIO_ENDPOINT
# 正确示例：http://minio:9000
# 错误示例：minio:9000（缺少协议前缀）
```

## 3. 数据库连接问题

### 3.1 连接池耗尽

**症状**：
```
HikariPool-1 - Connection is not available, request timed out after 30000ms
```

**排查与解决**：
```bash
# 1. 查看当前连接池状态（通过 Actuator）
curl http://localhost:8080/api/actuator/metrics/hikaricp.connections.active

# 2. 查看 MySQL 当前连接数
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} -e "
SHOW STATUS WHERE Variable_name = 'Threads_connected';
SHOW PROCESSLIST;
"

# 3. 临时增大连接池（在 application.yml 中）
# spring.datasource.hikari.maximum-pool-size: 50  （默认20）

# 4. 排查是否有未关闭的事务（查找 SLEEP 状态的连接）
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} -e "
SELECT id, user, host, db, command, time, state
FROM information_schema.processlist
WHERE command = 'Sleep' AND time > 60
ORDER BY time DESC;
"

# 5. 若有长时间占用连接的进程，终止它
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} -e "KILL <进程ID>;"
```

### 3.2 乐观锁冲突（笔记并发编辑）

**症状**：
```
OptimisticLockException: Failed to update record. Version mismatch: expected=3, actual=4
```

**说明**：这是正常的业务异常，表示两个用户同时编辑了同一笔记  
**处理方式**：
```java
// Service 层捕获并转换为 BusinessException
@Transactional
public NoteVO updateNote(Long noteId, UpdateNoteDTO dto, Long userId) {
    try {
        // ... 更新逻辑
        noteMapper.updateById(note);
    } catch (OptimisticLockException e) {
        throw new BusinessException("笔记已被他人修改，请刷新后重试");
    }
}
```

**客户端处理**：收到 `BusinessException` 时（HTTP 409）提示用户刷新后重试。

## 4. JWT 认证问题

### 4.1 Token 解析失败

**症状**：
```
HTTP 401 Unauthorized
{"code": 401, "message": "Token 无效或已过期"}
```

**常见原因与解决方案**：

```bash
# 原因1：Token 已过期（默认 24 小时）
# 解决：重新登录获取新 Token
curl -X POST http://localhost:8080/api/auth/login \
    -H "Content-Type: application/json" \
    -d '{"username":"user1","password":"pass123"}'

# 原因2：Authorization Header 格式错误
# 正确格式：Bearer {token}（注意有空格）
# ❌ 错误：Authorization: {token}
# ✓  正确：Authorization: Bearer eyJhbGci...

# 原因3：JWT_SECRET 与 Token 生成时不同（重启后 SECRET 变化）
# 解决：确保 JWT_SECRET 固定，不每次随机生成

# 原因4：Token 已在 Redis 黑名单中（用户已登出）
# 检查 Redis 黑名单
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} \
    KEYS "token:blacklist:*"
```

### 4.2 权限不足（403 Forbidden）

**症状**：
```
HTTP 403 Forbidden
{"code": 403, "message": "权限不足"}
```

**排查步骤**：
```bash
# 1. 确认用户角色
# 解码 JWT Token（在 https://jwt.io/ 工具中查看 claims）
# 或通过接口查看当前用户信息
curl http://localhost:8080/api/users/me \
    -H "Authorization: Bearer $TOKEN"

# 2. 确认接口需要的角色
# 查看 Controller 上的 @PreAuthorize 注解
# 例如：@PreAuthorize("hasRole('TEACHER')") 只允许 TEACHER 访问

# 3. 检查用户是否被禁用
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} \
    course_buddy -e "SELECT id, username, role, enabled FROM users WHERE username='user1';"
```

### 4.3 跨域（CORS）错误

**症状**：
```
Access to XMLHttpRequest at 'http://localhost:8080/api/...' from origin 'http://localhost:3000'
has been blocked by CORS policy
```

**解决方案**（在 SecurityConfig 中配置 CORS）：
```java
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOriginPatterns(List.of("http://localhost:3000", "https://your-domain.com"));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
}
```

## 5. WebSocket 问题

### 5.1 WebSocket 握手失败

**症状**：
```
WebSocket connection to 'ws://localhost:8080/api/ws/collaboration/1?token=...' failed:
Error during WebSocket handshake: Unexpected response code: 401
```

**解决方案**：
```bash
# 1. 确认 token 参数非空且未过期
# WebSocket URL 格式：ws://host/api/ws/collaboration/{id}?token={jwt_token}

# 2. 确认 Nginx WebSocket 代理配置正确
# nginx.conf 中必须有：
# proxy_http_version 1.1;
# proxy_set_header Upgrade $http_upgrade;
# proxy_set_header Connection "upgrade";

# 3. 测试 WebSocket 连接（使用 wscat 工具）
npm install -g wscat
wscat -c "ws://localhost:8080/api/ws/collaboration/1?token=${TOKEN}"
```

### 5.2 WebSocket 连接频繁断开

**症状**：客户端反复重连，日志显示 `Connection closed: 1006 (Abnormal Closure)`

**解决方案**：
```bash
# 1. 确认 Nginx 设置了足够长的超时时间
# proxy_read_timeout 3600s;
# proxy_send_timeout 3600s;

# 2. 客户端实现心跳机制
# 每 30 秒发送一次 PING 消息

# 3. 检查 Spring WebSocket 配置
# spring.websocket.max-session-idle-timeout=3600000  # 1小时
```

## 6. MinIO 文件上传问题

### 6.1 文件上传失败（大文件）

**症状**：
```
HTTP 413 Request Entity Too Large
```

**解决方案**：
```nginx
# nginx.conf 中增大允许的请求体大小
client_max_body_size 500m;  # 允许最大 500MB 的上传
```

```yaml
# application.yml 中设置 Spring 的文件大小限制
spring:
  servlet:
    multipart:
      max-file-size: 500MB
      max-request-size: 500MB
```

### 6.2 MinIO 连接超时

**症状**：
```
ConnectTimeoutException: connection timed out after 10000ms to /192.168.1.100:9000
```

**解决方案**：
```bash
# 1. 检查 MinIO 服务是否可达
curl -v http://${MINIO_ENDPOINT}/minio/health/live

# 2. 检查防火墙是否放行 9000 端口
sudo ufw allow 9000/tcp

# 3. Docker 网络检查
docker network inspect course-buddy_backend-network

# 4. 增大 MinIO 客户端超时
# 在 MinioConfig.java 中配置
MinioClient.builder()
    .endpoint(endpoint)
    .credentials(accessKey, secretKey)
    .httpClient(OkHttpClient.Builder()
        .connectTimeout(30, TimeUnit.SECONDS)
        .readTimeout(60, TimeUnit.SECONDS)
        .writeTimeout(60, TimeUnit.SECONDS)
        .build())
    .build();
```

### 6.3 分片上传卡住（文件不完整）

**症状**：分片上传初始化成功，但合并时报错文件不完整

**解决方案**：
```bash
# 查询 file_uploads 表中 status='UPLOADING' 的记录
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} \
    course_buddy -e "
SELECT id, original_name, upload_id, created_at
FROM file_uploads
WHERE status = 'UPLOADING'
    AND created_at < DATE_SUB(NOW(), INTERVAL 1 DAY)
ORDER BY created_at DESC;"

# 清理 MinIO 中未完成的分片上传
docker exec course-buddy-minio \
    mc alias set local http://localhost:9000 ${MINIO_ACCESS_KEY} ${MINIO_SECRET_KEY}
docker exec course-buddy-minio \
    mc incomplete remove local/course-buddy
```

## 7. Redis 缓存问题

### 7.1 Redis 内存不足

**症状**：
```
OOM command not allowed when used memory > 'maxmemory'
```

**解决方案**：
```bash
# 1. 查看 Redis 内存使用
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} INFO memory | grep used_memory_human

# 2. 清理过期 Key
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} DEBUG SLEEP 0

# 3. 增大 maxmemory（修改 docker-compose.yml 或 redis.conf）
# --maxmemory 512mb

# 4. 手动清理 Token 黑名单（仅过期的）
# Redis 的 TTL 机制会自动清理，无需手动操作

# 5. 查看最占内存的 Key
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} \
    --bigkeys
```

### 7.2 缓存雪崩/穿透

**排查步骤**：
```bash
# 查看 Redis 命中率
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} \
    INFO stats | grep keyspace_hits
# keyspace_hits 与 keyspace_misses 的比值即命中率
# 正常情况下命中率应 > 80%

# 查看慢查询日志
docker exec course-buddy-redis redis-cli -a ${REDIS_PASSWORD} \
    SLOWLOG GET 10
```

## 8. 性能问题

### 8.1 API 响应慢

**排查步骤**：
```bash
# 1. 查看慢 SQL（MySQL 慢查询日志）
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} -e "
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过1秒的查询记录为慢查询
SHOW GLOBAL VARIABLES LIKE 'slow_query_log_file';"

# 查看慢查询日志
docker exec course-buddy-mysql \
    tail -100 /var/log/mysql/mysql-slow.log

# 2. 使用 EXPLAIN 分析慢查询
docker exec course-buddy-mysql mysql -u root -p${MYSQL_ROOT_PASSWORD} -e "
EXPLAIN SELECT * FROM courses
WHERE status='PUBLISHED' AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 10;"

# 3. 查看 Actuator 指标
curl http://localhost:8080/api/actuator/metrics/http.server.requests | \
    python3 -m json.tool
```

## 9. 日志分析指南

### 9.1 日志文件位置

```bash
# 应用日志（Docker 环境）
docker logs course-buddy-app --tail 200

# 挂载的日志文件
ls -la $(docker inspect course-buddy-app \
    --format '{{range .Mounts}}{{if eq .Destination "/app/logs"}}{{.Source}}{{end}}{{end}}')

# Nginx 访问日志
docker exec course-buddy-nginx cat /var/log/nginx/access.log | tail -50

# Nginx 错误日志
docker exec course-buddy-nginx cat /var/log/nginx/error.log | tail -50
```

### 9.2 常用日志过滤命令

```bash
# 过滤错误日志
docker logs course-buddy-app 2>&1 | grep -i "ERROR\|FATAL\|Exception"

# 过滤特定用户的操作
docker logs course-buddy-app 2>&1 | grep "userId: 1001"

# 过滤慢请求（超过 1 秒）
docker logs course-buddy-app 2>&1 | grep "duration=[0-9]\{4,\}"

# 统计错误次数（按错误类型）
docker logs course-buddy-app 2>&1 | grep "ERROR" | \
    sed 's/.*ERROR \(.*\): .*/\1/' | sort | uniq -c | sort -rn | head -20

# 查看 Flyway 迁移日志
docker logs course-buddy-app 2>&1 | grep -i "flyway"
```

### 9.3 日志级别动态调整

```bash
# 通过 Actuator 动态修改日志级别（无需重启）
# 查看当前日志级别
curl http://localhost:8080/api/actuator/loggers/com.example.coursebuddybackend

# 临时开启 DEBUG 日志（排查问题用）
curl -X POST http://localhost:8080/api/actuator/loggers/com.example.coursebuddybackend \
    -H "Content-Type: application/json" \
    -d '{"configuredLevel": "DEBUG"}'

# 问题解决后恢复 INFO 级别
curl -X POST http://localhost:8080/api/actuator/loggers/com.example.coursebuddybackend \
    -H "Content-Type: application/json" \
    -d '{"configuredLevel": "INFO"}'
```

## 10. 容器问题

### 10.1 容器 OOM 被杀死（Out of Memory）

**症状**：
```bash
docker inspect course-buddy-app --format='{{.State.OOMKilled}}'
# 返回 true 表示被 OOM 杀死
```

**解决方案**：
```bash
# 1. 查看容器内存使用
docker stats course-buddy-app --no-stream

# 2. 增大容器内存限制（docker-compose.yml）
# deploy:
#   resources:
#     limits:
#       memory: 2g

# 3. 调整 JVM 内存参数
# JAVA_OPTS="-Xmx1g -Xms512m -XX:+UseContainerSupport"
```

### 10.2 健康检查持续失败

**症状**：`docker ps` 显示容器状态为 `(unhealthy)`

**排查步骤**：
```bash
# 查看健康检查执行历史
docker inspect --format='{{json .State.Health.Log}}' course-buddy-app | \
    python3 -c "import sys,json; [print(l['Output']) for l in json.load(sys.stdin)]"

# 手动执行健康检查命令
docker exec course-buddy-app \
    wget -qO- http://localhost:8080/api/actuator/health

# 常见原因：应用正在启动中（start_period 未过）
# 检查应用是否已完全启动
docker logs course-buddy-app 2>&1 | \
    grep "Started CoursebuddyBackendApplication"
```

## 11. 紧急恢复步骤

### 11.1 数据库崩溃恢复

```bash
# 1. 停止应用（避免写入损坏数据）
docker compose stop app

# 2. 检查 MySQL 状态
docker compose logs mysql | tail -50

# 3. 从最新备份恢复
# （假设有每日备份）
LATEST_BACKUP=$(ls -t backups/*.sql.gz | head -1)
echo "使用备份：$LATEST_BACKUP"

docker compose stop mysql
docker volume rm course-buddy-backend_mysql_data
docker compose up -d mysql

# 等待 MySQL 启动
sleep 30

# 恢复备份
gunzip -c $LATEST_BACKUP | \
    docker compose exec -T mysql \
    mysql -u root -p${MYSQL_ROOT_PASSWORD} course_buddy

# 4. 重启应用
docker compose up -d app
```

### 11.2 全服务重启

```bash
# 优雅重启所有服务
docker compose restart

# 强制重建并重启（有配置变更时）
docker compose down
docker compose up -d --build

# 仅重启应用（不影响数据库和缓存）
docker compose restart app
```
