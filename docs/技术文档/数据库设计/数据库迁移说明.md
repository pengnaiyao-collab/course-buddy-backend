# 数据库迁移说明

## 1. 概述

Course Buddy Backend 使用 **Flyway** 进行数据库版本管理。Flyway 是一款基于约定优于配置原则的数据库迁移工具，通过版本化的 SQL 脚本确保所有环境（开发、测试、生产）的数据库结构保持一致。

---

## 2. Flyway 配置

### 2.1 Maven 依赖

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-mysql</artifactId>
</dependency>
```

### 2.2 application.yml 配置

```yaml
spring:
  flyway:
    enabled: true
    # 迁移脚本目录（相对于 classpath）
    locations: classpath:db/migration
    # 基准版本（从此版本开始追踪，不执行更早的脚本）
    baseline-on-migrate: true
    baseline-version: 0
    # 允许乱序执行（开发环境建议开启，生产环境关闭）
    out-of-order: false
    # 迁移失败是否自动修复
    repair-on-migrate: false
    # 连接数据库的 Schema
    schemas: ${DB_NAME:course_buddy}
    # Flyway 元数据表名（默认 flyway_schema_history）
    table: flyway_schema_history
    # 是否验证已执行迁移文件的 checksum
    validate-on-migrate: true
```

## 3. 迁移文件命名规范

### 3.1 命名格式

```
V{版本号}__{描述}.sql
```

- **V**：固定前缀，表示版本化迁移（Versioned Migration）
- **{版本号}**：版本号，支持整数或点分格式（如 `1`、`1.1`、`2.0.1`）
- **__**（双下划线）：版本号与描述的分隔符
- **{描述}**：下划线分隔的英文描述，说明本次迁移的内容
- **.sql**：文件扩展名

### 3.2 版本号规则

```
整数版本（推荐用于主要变更）：
V1、V2、V3 ... V15

点分版本（用于补丁或紧急修复）：
V15.1__add_missing_index.sql
V15.2__fix_column_type.sql

团队协作场景（避免版本冲突）：
V20240115.1__feature_a.sql
V20240115.2__feature_b.sql
```

### 3.3 描述命名规范

```
操作类型前缀（统一规范）：
create_*     → 创建新表
add_*        → 添加字段或索引
modify_*     → 修改字段类型或约束
drop_*       → 删除表或字段
insert_*     → 插入初始数据
update_*     → 更新存量数据

示例：
V16__add_avatar_url_to_users.sql
V17__create_course_reviews_table.sql
V18__modify_notes_content_to_longtext.sql
V19__insert_default_admin_user.sql
```

## 5. 创建新迁移脚本

### 5.1 操作步骤

**步骤 1**：确定版本号（在现有最高版本基础上递增）
```bash
# 查看当前最高版本
ls src/main/resources/db/migration/ | sort -V | tail -5
```

**步骤 2**：创建迁移文件
```bash
# 在项目根目录执行
touch src/main/resources/db/migration/V16__add_avatar_to_users.sql
```

**步骤 3**：编写迁移 SQL

```sql
-- V16__add_avatar_to_users.sql
-- 作者：张三
-- 日期：2024-02-01
-- 说明：为用户表添加头像URL字段

-- 添加字段
ALTER TABLE users
    ADD COLUMN avatar_url VARCHAR(500) DEFAULT NULL
        COMMENT '用户头像URL'
        AFTER full_name;

-- 添加索引（如需要）
-- ALTER TABLE users ADD INDEX idx_avatar_url (avatar_url);
```

**步骤 4**：验证迁移（启动应用或使用 Flyway Maven 插件）
```bash
# 使用 Flyway Maven 插件验证
mvn flyway:validate

# 或直接启动应用（Flyway 会自动执行新迁移）
mvn spring-boot:run
```

**步骤 5**：提交代码
```bash
git add src/main/resources/db/migration/V16__add_avatar_to_users.sql
git commit -m "feat(db): add avatar_url to users table (V16)"
```

### 5.2 迁移脚本编写注意事项

```sql
-- ✓ 正确：使用 IF NOT EXISTS / IF EXISTS，幂等操作
ALTER TABLE users
    ADD COLUMN IF NOT EXISTS avatar_url VARCHAR(500) DEFAULT NULL;

-- ✓ 正确：添加索引时检查是否存在（MySQL 8.0+）
CREATE INDEX IF NOT EXISTS idx_users_avatar ON users(avatar_url);

-- ✓ 正确：数据迁移前备份
CREATE TABLE users_backup_20240201 AS SELECT * FROM users;

-- ✗ 错误：不要在迁移文件中 DROP 重要表（数据无法恢复）
-- DROP TABLE users;  ← 禁止

-- ✗ 错误：不要修改已执行的迁移文件（会导致 checksum 验证失败）
-- 已执行的文件必须保持不变

-- ✓ 正确：需要修改已执行的迁移时，创建新版本迁移
-- V16.1__fix_avatar_url_length.sql 来修正 V16 的问题
```

## 6. Flyway 元数据表

Flyway 将迁移历史记录在 `flyway_schema_history` 表中：

```sql
-- 查看迁移历史
SELECT
    installed_rank,
    version,
    description,
    type,
    script,
    checksum,
    installed_by,
    installed_on,
    execution_time,
    success
FROM flyway_schema_history
ORDER BY installed_rank;
```

**示例输出**：
```
+----------------+---------+------------------------------+------+--------------------------------+----------+--------------+---------------------+----------------+---------+
| installed_rank | version | description                  | type | script                         | checksum | installed_by | installed_on        | execution_time | success |
+----------------+---------+------------------------------+------+--------------------------------+----------+--------------+---------------------+----------------+---------+
|              1 | 1       | create users table           | SQL  | V1__create_users_table.sql     | -173xxx  | root         | 2024-01-01 10:00:00 |            245 |       1 |
|              2 | 2       | create courses table         | SQL  | V2__create_courses_table.sql   | -982xxx  | root         | 2024-01-01 10:00:01 |            189 |       1 |
...
+----------------+---------+------------------------------+------+--------------------------------+----------+--------------+---------------------+----------------+---------+
```

## 7. 常用 Flyway 命令

### 7.1 Maven 插件命令

```bash
# 查看迁移信息（不执行）
mvn flyway:info

# 执行所有待执行的迁移
mvn flyway:migrate

# 验证已执行迁移的 checksum 一致性
mvn flyway:validate

# 修复迁移历史（修复 checksum 不一致、删除失败记录）
mvn flyway:repair

# 清空数据库（仅开发环境使用！！！）
mvn flyway:clean

# 设置基准版本（将现有数据库标记为指定版本）
mvn flyway:baseline
```

**pom.xml 插件配置**：
```xml
<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <configuration>
        <url>jdbc:mysql://localhost:3306/course_buddy</url>
        <user>${DB_USER}</user>
        <password>${DB_PASSWORD}</password>
        <locations>
            <location>classpath:db/migration</location>
        </locations>
    </configuration>
</plugin>
```

### 7.2 Docker 环境中执行迁移

```bash
# 进入运行中的容器执行迁移
docker exec -it course-buddy-app mvn flyway:migrate

# 或通过环境变量连接数据库
mvn flyway:info \
  -Dflyway.url=jdbc:mysql://localhost:3306/course_buddy \
  -Dflyway.user=app_user \
  -Dflyway.password=app_password
```

## 8. 回滚策略

### 8.1 重要说明

Flyway 社区版**不支持自动回滚**，需要通过以下方式手动处理：

### 8.2 方案一：撤销迁移（推荐）

创建新的迁移脚本来撤销变更：

```sql
-- V16__add_avatar_to_users.sql（原始迁移）
ALTER TABLE users ADD COLUMN avatar_url VARCHAR(500) DEFAULT NULL;

-- V17__revert_avatar_from_users.sql（撤销脚本）
ALTER TABLE users DROP COLUMN avatar_url;
```

### 8.3 方案二：数据库快照恢复

```bash
# 部署前备份（强烈推荐）
mysqldump -h localhost -u root -p course_buddy > backup_before_v16.sql

# 回滚时恢复备份
mysql -h localhost -u root -p course_buddy < backup_before_v16.sql

# 然后从 flyway_schema_history 删除对应记录
DELETE FROM flyway_schema_history WHERE version >= '16';
```

### 8.4 方案三：修复失败的迁移

```bash
# 迁移失败后修复（删除失败记录，允许重新执行）
mvn flyway:repair

# 修正 SQL 后重新执行
mvn flyway:migrate
```

## 9. 团队协作规范

```
团队多人开发时的迁移管理规范：

1. 每个功能分支只创建一个迁移文件
2. 合并前检查版本号是否与主分支冲突
3. 使用日期格式版本号避免冲突（V20240201.1__xxx.sql）
4. 迁移文件一旦推送到主分支，禁止修改（只能新增撤销脚本）
5. 每次迁移前在本地先执行 mvn flyway:validate 验证
6. CI/CD 管道中的迁移应在应用启动前执行（Init Container 或 Flyway Maven 插件）
7. 生产环境迁移前必须先在测试环境验证
```

## 10. 常见问题

### Q1：启动时报 "Found more than one migration with version X"

**原因**：两个迁移文件使用了相同的版本号  
**解决**：检查 `db/migration/` 目录，修改其中一个的版本号

### Q2：报 "Validate failed: Migration checksum mismatch"

**原因**：已执行的迁移文件内容被修改（Flyway 校验 checksum）  
**解决**：
```bash
# 方案1：恢复文件到原始内容
git checkout src/main/resources/db/migration/VX__xxx.sql

# 方案2（仅开发环境）：修复 checksum
mvn flyway:repair
```

### Q3：如何在测试时重置数据库

```yaml
# application-test.yml（测试环境）
spring:
  flyway:
    clean-on-validation-error: true  # 验证失败时清空重建
  sql:
    init:
      mode: always
```
